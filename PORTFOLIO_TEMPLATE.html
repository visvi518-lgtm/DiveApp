Overview

기획 문서부터 시작해서, 검증 가능한 것에 집중

프리다이버, 스쿠버다이버, 다이빙 강사·교육기관을 대상으로 실제 서비스 출시를 전제로 기획. 요구사항 정의 → 유저플로우 → DB 스키마 → API 설계 → 화면 설계 순서로 문서를 먼저 작성한 뒤 구현에 들어가는 Documentation-First 워크플로우 진행.

Pivot decision

원래 계획은 iOS(SwiftUI) 우선 개발. iOS 파운데이션 구현까지 마친 시점에 개발 환경에 Mac/Xcode가 없어 실제 컴파일 검증이 불가능하다는 제약이 드러났고, "검증 가능한 것부터 완성도를 높인다"는 원칙에 따라 iOS는 보류하고 Android(Kotlin/Compose)와 Web(React)을 우선 진행하는 방향으로 전환. 이 판단 덕분에 이후 모든 기능을 실제 에뮬레이터·브라우저·로컬 백엔드로 끝까지 검증.

Problem solving

구현보다 오래 걸린 건, "진짜로 동작하는지" 확인하는 과정

아래 7건은 전부 오프라인·목(mock) 기반 테스트로는 원천적으로 발견할 수 없고, 실제 DB·실제 기기·실제 계정으로 전체 플로우를 끝까지 돌려봐야만 드러난 문제.

SESSION A

로컬 PostgreSQL + 실제 백엔드 연동

오프라인 SQL 생성과 목 기반 스모크 테스트만 하던 단계에서, 처음으로 진짜 DB에 마이그레이션을 실행하고 실제 기기를 백엔드에 붙였다.

Alembic 마이그레이션의 Enum 타입 중복 생성

증상
DuplicateObjectError — 마이그레이션 실행 즉시 실패
원인
Enum 타입을 create_table() 이전에 미리 생성했는데, create_table()이 컬럼용으로 같은 타입을 자동으로 또 생성 시도. 오프라인 SQL 생성(--sql)은 실제 실행이 없어 이 문제가 드러나지 않았음
해결
불필요한 사전 생성 로직 제거

Timezone-naive datetime 컬럼

증상
배너 목록 조회 API에서 500 에러
원인
5개 모델의 날짜 컬럼이 DateTime(timezone=True) 명시 없이 선언되어, timezone-aware 값과 비교하는 쿼리에서 타입 불일치. 마이그레이션 DDL 자체는 처음부터 TIMESTAMPTZ로 올바르게 정의되어 있었고 ORM 모델 코드만 어긋나 있었음
해결
5개 모델 파일에 timezone=True 추가, DB 재생성 불필요

Android 9+ 평문 HTTP 통신 차단

증상
에뮬레이터의 모든 API 요청이 조용히 실패
원인
Network Security Config 부재로 CLEARTEXT communication not permitted
해결
디버그 빌드 전용 Network Security Config 추가 (릴리즈 빌드는 HTTPS라 영향 없음)
SESSION B

실제 Naver·Google 계정으로 로그인 완주

실제 OAuth 앱을 등록하고, 처음으로 진짜 계정으로 소셜 로그인을 끝까지 시도했다.

Naver 로그인 콜백 페이지의 .init() 누락

증상
실제 로그인에 성공해도 항상 "로그인이 완료되지 않았습니다"
원인
콜백 페이지가 SDK 인스턴스를 생성만 하고 .init()을 호출하지 않아, SDK가 URL의 OAuth 응답을 절대 파싱하지 못함. 실제 SDK 소스를 직접 읽어 init() 내부의 callbackHandler → oauthCallback() → accessToken 흐름을 확인해 원인 특정
해결
.init() 호출과 필수 DOM 컨테이너 추가

서로 다른 Google OAuth Client ID

증상
구글 로그인 시도 시 항상 401 (로그인 만료 메시지)
원인
Backend·Web 환경변수 파일에 실수로 서로 다른 Client ID가 채워져 있어 ID 토큰의 aud 클레임이 백엔드 기대값과 불일치
해결
두 값을 하나로 통일
수정 후 — 실제 Naver 계정으로 완주
수정 후 — 실제 Google 계정으로 완주
SESSION C

이메일/비밀번호 인증 추가

회원가입 → 프로필 설정 → 홈 → 로그아웃 → 재로그인까지, 처음으로 전체 사이클을 실제 브라우저에서 클릭해보다가 발견 — 둘 다 원래 소셜 로그인에도 있던 기존 버그였다.

프로필 설정 화면에 라우트 가드 없음

증상
저장에 성공해도 계속 같은 화면에 머무름
원인
저장 API 성공 후 다른 화면으로 이동시키는 로직 자체가 없었음
해결
authenticated 상태가 되면 자동으로 홈으로 이동하는 라우트 가드 추가

로그인 화면이 "프로필 설정 필요" 상태 미처리

증상
신규가입 직후에도 로그인 화면에 그대로 남아있음
원인
로그인 화면의 라우트 가드가 authenticated 상태만 확인하고 needsProfileSetup은 처리하지 않았음
해결
해당 상태일 때 프로필 설정 화면으로 이동하는 조건 추가
Web — 실제 등록된 다이빙 기록
Android — CO₂ Table 타이머 실시간 동작

회원 관리, 정보글/배너 CRUD, 서비스 통계를 제공하는 관리자 화면을 별도 앱 없이 기존 Web 서비스 안에 역할 기반으로 통합했고, 실제 계정을 관리자 권한으로 전환해 통계 조회부터 CRUD, 배너 노출 순서 중복 검증 로직까지 실제 백엔드로 확인.

오프라인·목 기반 테스트로는 원천적으로 발견할 수 없고,
