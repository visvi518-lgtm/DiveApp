# Report 09 — Admin 대시보드 웹 화면

## 배경

사용자가 지시한 순서("1번부터 쭉 진행해보자")의 3번째 항목. 백엔드 Admin API(Phase 2)는 이미 완성되어 있었지만 대응하는 웹 클라이언트가 없었다. 이번에는 기존 일반 사용자용 Web 앱(React + Vite) 안에 `/admin` 하위 라우트로 관리자 화면을 추가했다 — 별도 앱을 새로 만들지 않고 기존 앱을 재사용하는 쪽을 택했다 (같은 axios 클라이언트, 같은 인증 세션, `AppLayout`에 role 기반으로 "관리자" 메뉴만 조건부 노출).

---

## 구현 범위

백엔드 Admin API가 실제로 지원하는 3개 도메인만 구현했다 (Community/DiveLog/Training/Certificate에는 Admin API 자체가 없어 대상 아님):

1. **대시보드** (`/admin`) — 총 회원/오늘 신규가입/활성 회원/게시글/댓글/다이브로그 수 6개 통계 카드 + 하위 메뉴 진입 링크
2. **회원 관리** (`/admin/users`, `/admin/users/:id`) — 이메일/닉네임 검색, 목록, 상세(가입 경로/상태/가입일/마지막 로그인), 계정 정지/정지 해제
3. **정보 게시판 관리** (`/admin/information`, `/new`, `/:id/edit`) — 발행/비공개 여부 포함 전체 목록, 작성/수정(제목·내용·썸네일·발행여부), 삭제
4. **배너 관리** (`/admin/banners`, `/new`, `/:id/edit`) — 노출 순서순 정렬 목록, 작성/수정(제목·이미지 URL·유형·연결 URL·노출 순서·활성 여부·노출 기간), 삭제

접근 제어는 기존 `RequireAuth` 패턴과 동일한 스타일로 `RequireAdmin`을 추가해 `currentUser?.role !== 'ADMIN'`이면 홈으로 리다이렉트한다. 실제 인가는 백엔드의 `require_admin` 의존성이 최종 방어선이며, 이건 Report 02에서 이미 검증된 기존 로직이라 이번에 다시 확인하지 않았다.

---

## 설계상 유의점

### 백엔드에 없는 단건 조회(GET /{id})
정보글/배너 admin 라우터는 목록/생성/수정/삭제만 있고 단건 조회가 없다. 수정 화면 진입 시 목록 페이지에서 이미 갖고 있는 객체를 React Router의 `navigate(path, { state })`로 그대로 넘겨 채우고, 직접 URL 진입/새로고침처럼 state가 없는 경우에만 전체 admin 목록을 다시 불러와 id로 찾는 방식으로 처리했다. 데이터가 많아지면 비효율적인 임시방편이므로, 백엔드에 단건 조회 엔드포인트를 추가하는 게 장기적으로는 더 나은 설계라는 점을 백로그에 기록해뒀다.

### Admin 회원 응답에 `role` 필드 없음
`AdminUserListItem`/`AdminUserDetailResponse` 둘 다 `role` 필드를 반환하지 않는다 (이메일/가입경로/상태/닉네임/가입일 정도). 그래서 "이 회원이 이미 관리자인지" 표시하거나 역할을 바꾸는 기능은 만들 수 없었다 — 백엔드 스키마 확장이 필요하면 별도로 결정할 사안이라 백로그에만 남겼다.

## 실제 검증

로컬 백엔드(Report 07/08에서 띄운 것과 동일 세션, 여전히 살아있었음)에 대해 실제로 확인했다:

1. `npm run build` — 클린 (TS 컴파일 에러 없음). 진행 중 `AppLayout`의 nav 아이템 배열에 관리자 메뉴를 조건부로 추가하면서 유니온 타입 불일치로 컴파일 에러가 한 번 났고, 타입을 명시해서 해결했다.
2. 기존 테스트 계정(`tester@example.com`)을 DB에서 직접 `role=ADMIN`으로 승격시키고 새 JWT를 발급받아, headless Chrome + puppeteer-core로 실제 화면을 클릭하며 확인:
   - 로그인 후 상단 네비게이션에 "관리자" 메뉴가 실제로 나타남 (일반 유저였다면 안 보였을 항목)
   - 대시보드 통계 카드가 실제 DB 값(총 회원 1, 게시글 1, 다이브로그 1 등)을 정확히 표시
   - 회원 상세 페이지가 실제 이메일/가입경로/가입일을 표시
   - 정보글 작성 → 실제로 DB에 저장되고 목록에 "비공개 · 조회 0"으로 표시됨을 확인
   - 배너 작성 시 **실제 유니크 제약 위반**(`display_order 1 is already in use`, 이전 세션에서 이미 존재하던 배너와 충돌)이 그대로 화면에 표시되는 것을 확인 — 크래시 없이 백엔드 검증 메시지가 정확히 전달됨
   - 다른 노출 순서 값으로 재시도 → 정상 저장되고 목록으로 리다이렉트됨을 확인
   - 정보글/배너 모두 삭제 버튼 클릭(확인 다이얼로그 accept) → 목록에서 실제로 사라지는 것까지 확인 (삭제 전/후 개수 비교로 검증)
3. 테스트에 사용한 puppeteer-core, 임시 스크립트, 임시 관리자 승격 스크립트는 모두 삭제. 테스트 중 생성한 정보글/배너 테스트 데이터도 삭제 테스트를 겸해 정리했다 (배너 1건은 이전 세션에서 남아있던 것이라 그대로 둠 — 이 로컬 DB는 임시 개발용이라 문제 없음).
